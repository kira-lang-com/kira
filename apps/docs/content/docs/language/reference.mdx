## Declarations

The current frontend recognizes these top-level declaration forms:

- `import`
- `construct`
- `type`
- `function`
- construct-defined forms such as `Widget Dashboard(...) { ... }`

Examples from the repo:

```ts
import bindings.callbacks as callbacks

@Main
@Native
function main() {
    let result: I64 = callbacks.kira_invoke_callback(add_two, 0, 5)
    print(result)
    return
}
```

```ts
construct Widget {
    annotations {
        @State;
        @Binding;
        @Env;
        @Doc;
    }

    requires {
        content;
    }
}
```

## Builtin Types

These names are resolved by the current semantics layer:

| Group | Names |
| --- | --- |
| General | `Void`, `Int`, `Float`, `Bool`, `String` |
| Fixed-width integers | `I8`, `U8`, `I16`, `U16`, `I32`, `U32`, `I64`, `U64` |
| Native/FFI helpers | `RawPtr`, `CString`, `CBool` |

The compiler also works with named types declared in Kira source and with generated FFI types.

## FFI Type Annotations

Generated bindings and hand-written native interop use these annotation families today:

- `@FFI.Extern`
- `@FFI.Callback`
- `@FFI.Pointer`
- `@FFI.Struct`
- `@FFI.Alias`
- `@FFI.Array`

Those are not decorative annotations. They carry real compiler semantics for the current FFI path.

## Function Rules

Current semantic rules worth knowing:

- `@Main` must appear exactly once
- `@Main` functions must have a valid entry signature
- parameters require explicit types
- local `let` bindings can infer types only when the inference is unambiguous
- literal coercion is allowed in some explicit typed contexts, for example assigning an integer literal to a `Float`

This fails today:

```ts
@Main
function entry() {
    let value;
    return;
}
```

The repo's semantics tests expect that to raise `KSEM029` ("type inference is ambiguous").

## Expressions And Control Flow

The frontend understands these expression families:

- integer, float, string, and boolean literals
- arrays
- unary and binary operators
- grouped expressions
- member access
- namespaced references
- call syntax

It also parses and models these control-flow forms:

- `if`
- `for`
- `switch`

Important boundary:

- `check`, `tokens`, and `ast` can operate on that broader surface
- `run` and `build` still require the currently lowered executable subset

That is why examples such as `examples/complex_language_showcase` are useful frontend proofs without also being runnable backend proofs.

## Statement Style

The checked-in Kira examples use both:

- semicolon-terminated statements
- newline-delimited statements

Both styles are accepted by the current parser in the forms used across the repo examples.

## Print And Current Backend Limits

The VM runtime can format runtime values such as integers, strings, booleans, and raw pointers.

For the fully proven cross-backend contract, the repo currently demonstrates `print(...)` with:

- integers
- strings

Native lowering is still stricter than the full frontend/type system, so do not assume every typed expression can be printed or lowered just because it parses cleanly.
