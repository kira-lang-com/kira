## Entry Syntax

The working bootstrap syntax uses `function` declarations plus an explicit `@Main` annotation:

```ts
@Main
function entry() {
    let x = 1 + 2;
    print(x);
    print("hello from kira");
    return;
}
```

The entrypoint is chosen by `@Main`, not by the function name.

## Core Execution Annotations

Kira currently gives compiler meaning to three execution annotations:

- `@Main`: marks the single entrypoint
- `@Runtime`: keeps a function in bytecode/runtime execution
- `@Native`: compiles a function for native execution

Those annotations matter differently depending on the backend:

- VM requires runtime execution only
- LLVM/native needs a native-capable program
- hybrid can mix runtime and native functions in one module

## Frontend Surface vs Executable Surface

This repo already has a broader frontend than the current lowering boundary.

The frontend understands, checks, or preserves:

- `import`, `construct`, `type`, and `function`
- annotations, including namespaced annotations and annotation arguments
- construct sections such as `annotations`, `requires`, `lifecycle`, `builder`, and `representation`
- control-flow syntax like `if`, `for`, and `switch`
- builder/content blocks and construct-defined declaration forms
- lifecycle hooks such as `onAppear()` and `onChange(of: value)`

The current executable pipeline is intentionally smaller. The repo explicitly proves these as runnable today:

- `@Main`, `@Runtime`, `@Native`
- `function`
- integer and string literals
- local `let`
- identifier loads
- integer `+`
- `return`
- block statements
- direct function calls with arguments/results in the lowered scalar and pointer subset
- builtin `print` for the proven integer and string cases
- explicit FFI extern calls
- generated callback, pointer, alias, array, and struct types used by the FFI path

## Readable First

The repo's examples show the language style Kira is aiming for:

- direct names
- explicit annotations when behavior changes materially
- inference on locals when the compiler can prove the type without guessing
- library-defined language extensions built on stable compiler mechanisms

That is why the compiler accepts richer construct-driven examples such as `examples/complex_language_showcase` even though the runnable backend subset is smaller.

## What Belongs To Kira Libraries

Kira's direction is not "put all high-level behavior in Zig."

The current compiler does these jobs:

- parse and model constructs
- validate annotation use
- validate lifecycle hooks and required content blocks
- lower the currently supported executable subset

Higher-level framework behavior still belongs in Kira libraries once those language hooks exist. The checked-in construct examples are demonstrating that boundary, not erasing it.
